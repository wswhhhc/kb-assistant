# 企业知识库智能助手数据库设计文档 V1

## 1. 文档说明

本文档用于描述《企业知识库智能助手》项目的数据库设计方案，明确关系型数据库 MySQL 的表结构设计、字段定义、约束规则、索引设计、状态值规范及与向量数据库 Qdrant 的数据映射关系，为后续开发、建表、联调和维护提供依据。

## 2. 设计目标

数据库设计目标如下：

- 支撑用户、角色、知识库、成员、文档、问答、反馈等核心业务
- 保证业务数据结构清晰，便于后续扩展
- 满足知识库成员隔离与权限控制要求
- 便于与向量数据库配合实现 RAG 检索
- 满足分页查询、统计分析、日志追踪等常见需求

## 3. 数据库选型

关系型数据库：`MySQL 8.x`

选型原因：

- 生态成熟，易于与 Spring Boot 集成
- 适合存储结构化业务数据
- 支持索引、事务、约束，便于保证数据一致性
- 适合做后台管理系统的数据支撑

向量数据库：`Qdrant`

用途说明：

- 存储文档切片向量
- 支持相似度检索
- 支持按知识库、文档等条件过滤

缓存数据库：`Redis`

用途说明：

- 缓存登录态
- 缓存热点数据
- 缓存临时处理状态

## 4. 关系型数据库设计原则

- 所有主表均使用 `BIGINT` 作为主键 ID
- 时间字段统一采用 `DATETIME`
- 逻辑状态字段采用 `VARCHAR` 或 `TINYINT`
- 涉及唯一性的业务字段增加唯一索引
- 频繁查询的外键字段增加普通索引
- 删除高关联数据时优先采用业务控制删除，不完全依赖数据库级联
- **V1 索引策略**：优先保证主键、唯一索引和外键索引，普通索引按需添加。V1 数据量较小（预计万级），不宜为所有查询字段都建索引，避免拖慢写入性能。可在 V2 根据慢查询日志补充索引。

## 5. 核心实体关系

核心实体包括：

- 用户
- 知识库
- 知识库成员
- 文档
- 文档切片
- 会话
- 消息
- 答案反馈
- 失败问题

实体关系说明：

- 一个用户可加入多个知识库
- 一个知识库可包含多个成员
- 一个知识库可包含多个文档
- 一个文档可包含多个切片
- 一个用户可创建多个会话
- 一个会话可包含多条消息
- 一条回答可对应多条反馈记录中的一条当前有效记录

表说明：存储系统用户基础信息。

| 字段名 | 类型 | 不能为空 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | 是 |  | 主键 ID |

| role | VARCHAR(20) | 是 | `USER` | 角色，`ADMIN` 或 `USER` |

## 7. 表关系说明

主要逻辑关系如下：

- `sys_user.id` -> `knowledge_base.owner_user_id`
- `sys_user.id` -> `knowledge_base_member.user_id`
- `knowledge_base.id` -> `knowledge_base_member.knowledge_base_id`
- `knowledge_base.id` -> `document.knowledge_base_id`
- `document.id` -> `document_chunk.document_id`
- `knowledge_base.id` -> `document_chunk.knowledge_base_id`
- `sys_user.id` -> `chat_session.user_id`
- `knowledge_base.id` -> `chat_session.knowledge_base_id`
- `chat_session.id` -> `chat_message.session_id`
- `chat_message.id` -> `answer_feedback.message_id`
- `knowledge_base.id` -> `failed_question.knowledge_base_id`

说明：

- V1 可以在数据库层不强制设置全部外键约束，由应用层维护一致性
- 如需严格约束，可在后续版本视性能和维护要求增加外键

## 8. Qdrant 设计

### 8.1 Collection 设计

建议 collection 名称：`kb_chunks`

建议配置：

- vector size：`1024`
- distance：`Cosine`

说明：

- 若使用 `BGE-M3`，可采用 1024 维向量

### 8.2 Payload 字段设计

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| knowledge_base_id | integer | 所属知识库 ID |
| document_id | integer | 所属文档 ID |
| chunk_id | integer | 切片 ID |
| chunk_index | integer | 切片序号 |
| file_name | string | 文件名称 |
| page_no | integer | 页码 |
| content_preview | string | 切片摘要 |

说明：

- 检索时根据 `knowledge_base_id` 做过滤
- 检索结果根据 `chunk_id` 与 `document_chunk` 记录关联

## 10. 字段命名规范

- 表名统一使用小写英文加下划线
- 主键统一命名为 `id`
- 外键字段统一使用 `{实体名}_id`
- 时间字段统一使用 `created_at`、`updated_at`
- 状态字段统一使用具备明确业务语义的名称，如 `status`、`parse_status`

## 12. 数据一致性设计建议

- 知识库和文档表预留了 `is_deleted` 逻辑删除字段，支持误删恢复
- 逻辑删除的记录对业务不可见，物理删除由定时任务或管理员手动触发
- 文档物理删除时，应同步删除 `document_chunk` 和 Qdrant 中对应向量
- 知识库物理删除时，应同步删除成员关系、文档、切片、会话、失败问题及向量数据
- 会话删除时，应同步删除消息记录
- 反馈删除一般不做物理删除，可视业务需要保留
