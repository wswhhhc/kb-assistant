# 企业知识库智能助手项目开发计划文档 V1

## 1. 文档说明

本文档基于现有《需求文档》《系统设计文档》《数据库设计文档》编写，目标是为《企业知识库智能助手》项目提供一套可执行的开发推进计划。文档重点描述项目开发顺序、模块拆分、阶段目标、任务清单、交付物、验收标准和风险控制，便于按步骤完成整个项目。

本文档面向个人项目开发场景，强调以下原则：

- 先搭框架，再补功能
- 先跑通主链路，再做优化项
- 先完成后端核心能力，再完善前端交互
- 每一阶段都要形成可运行、可验证、可演示的成果

## 2. 项目开发目标

项目最终目标是完成一个可演示、可部署、可用于实习面试展示的企业知识库智能问答平台，满足以下能力：

- 支持管理员和普通用户登录
- 支持知识库管理和成员权限控制
- 支持文档上传、解析、切片、向量化、入库
- 支持基于知识库的智能问答
- 支持回答引用溯源
- 支持会话历史、答案反馈、失败问题记录
- 支持 Docker Compose 一键部署

## 3. 开发总原则

### 3.1 主线优先原则

优先完成以下主链路：

1. 登录
2. 知识库创建
3. 文档上传
4. 文档解析和向量入库
5. 问答检索
6. 答案生成
7. 引用返回

只有主链路稳定后，再继续做反馈、统计、优化等功能。

### 3.2 模块化推进原则

开发按模块逐步推进，每完成一个模块都要具备以下成果：

- 代码可运行
- 基本接口可调用
- 核心流程可验证
- 对应文档同步更新

### 3.3 接口契约先行原则

前后端并行开发时，建议在第二阶段初期先定义好接口契约（请求/响应结构、状态码、路径参数），可采用以下方式之一：

- 使用 Swagger / OpenAPI 生成接口文档
- 在前端侧维护一份接口定义清单

接口契约确定后，前后端可并行开发，减少联调阶段的返工。

### 3.4 面试导向原则

项目不以”功能越多越好”为目标，而以”结构清晰、功能闭环、能讲清楚设计思路”为目标。

### 3.5 技术选型统一说明

以下为项目关键技术决策，各阶段开发中应保持一致：

| 决策项 | 选型方案 | 说明 |
|--------|---------|------|
| 前端 UI 框架 | Element Plus | 组件齐全，文档成熟，不必自己写表格/弹窗/表单 |
| 后端 ORM | MyBatis-Plus | 代码生成 + 条件构造器效率高，适合个人开发 |
| 认证方案 | JWT | 无状态，登录后前端存本地，请求时带 Header |
| 文件传输方式 | 共享卷 + 文件路径 | Spring Boot 接收上传后保存到共享目录，将文件路径传给 FastAPI 读取；避免二进制流传输的复杂度 |
| 文件存储路径 | `./data/files/{kb_id}/{UUID}_{原始文件名}` | UUID 防重名，kb_id 分组便于管理 |
| 文档处理模式 | 同步处理 + 前端轮询 | Spring Boot 同步调用 FastAPI 处理接口（设 2 分钟超时），前端定期查状态；V1 不引入消息队列 |
| 切片策略 | V1 先用固定长度切片 | 500 字一段 + 50 字 overlap 兜底，标题感知切片作为后续优化 |
| Embedding 维度 | 待确认 SiliconFlow 输出维度 | 文档中暂写 1024，实际开发时确认后更新 Qdrant collection 配置 |

## 4. 总体阶段划分

整个项目建议分为 6 个阶段推进：

1. 项目初始化阶段（约 1-2 天）
2. 基础业务模块阶段（约 3-5 天）
3. 文档入库链路阶段（约 3-5 天）
4. RAG 问答链路阶段（约 3-5 天）
5. 闭环优化模块阶段（约 2-3 天）
6. 测试部署与项目包装阶段（约 3-4 天）

> 以上为个人全栈开发的粗略估算，实际节奏可根据个人熟悉程度调整。总工期预计 3-4 周。

## 5. 分阶段开发计划

## 5.1 第一阶段：项目初始化

### 阶段目标

完成项目骨架、开发环境、基础配置和目录结构搭建，使前后端及依赖服务能够启动。

### 开发内容

#### 5.1.1 创建项目目录结构

建议目录结构：

- `frontend/`：Vue 3 前端项目
- `backend-java/`：Spring Boot 业务服务
- `backend-python/`：FastAPI AI 服务
- `sql/`：数据库建表脚本
- `docker/`：Docker 相关配置
- `docs/`：需求、设计、计划类文档

#### 5.1.2 初始化前端项目

任务内容：

- 使用 `Vue 3` 创建前端项目，引入 Element Plus 作为 UI 组件库
- 接入路由（vue-router）
- 接入状态管理（Pinia）
- 配置基础页面布局
- 预留登录页、后台管理页、问答页路由

预期成果：

- 前端项目可正常启动
- 可进入基础页面框架

#### 5.1.3 初始化 Spring Boot 项目

任务内容：

- 创建 Spring Boot 工程
- 引入 Spring Web、MyBatis-Plus、MySQL Driver、Spring Data Redis、JWT（jjwt）等依赖
- 配置多环境配置文件
- 配置基础异常处理、统一响应体、接口前缀

预期成果：

- Spring Boot 服务可启动
- 可连接 MySQL 和 Redis

#### 5.1.4 初始化 FastAPI 项目

任务内容：

- 创建 FastAPI 工程
- 安装文档解析（PyMuPDF、python-docx）、模型调用（httpx/requests）、向量数据库客户端（qdrant-client）等依赖
- 配置基础路由结构
- 配置模型服务调用配置（SiliconFlow API Key、Endpoint、模型名、向量维度）
- 确认共享文件卷路径配置，与 Spring Boot 的 `./data/files/` 目录映射一致

预期成果：

- FastAPI 服务可启动
- 可访问基础健康检查接口

#### 5.1.5 初始化基础依赖服务

任务内容：

- 启动 MySQL
- 启动 Redis
- 启动 Qdrant
- 编写 Docker Compose 初版配置

预期成果：

- 本地可启动全部基础依赖
- 各服务网络可互通

### 本阶段交付物

- 前端项目骨架
- Spring Boot 项目骨架
- FastAPI 项目骨架
- Docker Compose 初版

### 本阶段验收标准

- 前端、Spring Boot、FastAPI 都能成功启动
- MySQL、Redis、Qdrant 可访问

## 5.3 第三阶段：文档入库链路开发

### 阶段目标

完成文档从上传到解析、切片、向量化、入库的完整流程，使知识库具备可检索数据。

### 开发内容

#### 5.3.1 文档上传模块

任务内容：

- 实现文件上传接口
- 文件落盘存储，路径格式 `./data/files/{knowledge_base_id}/{UUID}_{原始文件名}`
- 保存文档元数据到 document 表（文件名、路径、大小、类型等）
- 文档状态初始化为 `UPLOADED`

前端内容：

- 文档上传按钮
- 文档列表页
- 状态显示

预期成果：

- 管理员可向知识库上传文档

#### 5.3.2 文档管理模块

任务内容：

- 查询文档列表
- 查看文档状态
- 删除文档
- 查看 chunk 数量、失败原因等信息

预期成果：

- 文档管理后台可用

#### 5.3.3 文档解析模块

任务内容：

- FastAPI 实现 PDF 解析
- FastAPI 实现 DOCX 解析
- FastAPI 实现 Markdown 解析
- FastAPI 实现 TXT 解析
- 标准化输出解析结果

建议实现顺序：

1. TXT
2. Markdown
3. DOCX
4. PDF

PDF 解析边界说明：

- V1 只保证纯文本型 PDF 的正文提取
- 表格、多栏布局、扫描件 PDF 不做精确还原，提取结果可能混乱
- 面试时可说明 "这是 V1 的已知局限，后续可引入 OCR 或非结构化解析方案"

预期成果：

- 支持四类文档文本提取

#### 5.3.4 文本切片模块

任务内容：

- 按标题和自然段优先切分，保留章节结构
- 控制 chunk 长度约 300-500 中文字
- 相邻 chunk 保留 50-100 字 overlap
- V1 兜底策略：若无法识别标题结构，回退到固定长度切片（500 字符一段 + 50 字符 overlap）
- 保存 chunk 元数据（chunk_index、section_title、content、page_no、char_count、token_count）到 MySQL

预期成果：

- 生成结构合理的切片记录

#### 5.3.5 向量化模块

任务内容：

- 调用 SiliconFlow embedding 接口
- 生成文本向量
- 写入 Qdrant
- 更新 `document_chunk.vector_id`

预期成果：

- 每个 chunk 都有对应向量记录

#### 5.3.6 文档处理编排接口

任务内容：

- Spring Boot 同步调用 FastAPI 文档处理接口（`POST /internal/document/process`）
- 接口设置 2 分钟超时，大文档超时后标记为 `FAILED`
- 前端上传文档后定时轮询文档状态（每 3 秒一次），直到 `READY` 或 `FAILED`
- 维护文档状态流转（UPLOADED → PARSING → CHUNKING → EMBEDDING → READY / FAILED）
- 失败时记录错误原因到 `error_message` 字段

预期成果：

- 文档上传后可一键处理到 `READY`

### 本阶段交付物

- 文档上传模块
- 文档管理模块
- 文档解析模块
- 切片模块
- 向量化模块
- Qdrant 基础数据入库能力

### 本阶段验收标准

- 文档上传成功
- 至少一类文档能完成全流程处理
- Qdrant 中可看到向量数据
- MySQL 中有对应 chunk 元数据

## 9. 风险与应对方案

### 9.1 文档解析效果不稳定

风险说明：

- PDF 解析可能存在格式错乱
- 表格、多栏、扫描件 PDF 提取效果差

应对方案：

- 第一版优先支持结构简单文档
- 先保证 TXT、Markdown、DOCX 稳定
- PDF 以正文文本提取为主，不追求复杂版式还原
- 面试时主动说明此局限，体现对边界情况有意识

### 9.2 向量检索效果不理想

风险说明：

- 切片不合理会影响召回质量
- TopK 配置不合适可能导致答案质量下降

应对方案：

- 先使用简单稳定的切片策略
- 准备一组测试问题反复调试 chunk 长度和 TopK
- 后续引入 rerank 作为优化项

### 9.3 模型调用成本或稳定性问题

风险说明：

- 外部 API 可能有延迟或失败

应对方案：

- 增加异常重试和失败提示
- 对模型配置（API Key、Endpoint、向量维度等）做环境变量化
- **开发前确认 SiliconFlow 上 BGE-M3 的实际输出向量维度**，避免 Qdrant collection 配置与模型输出不匹配
- 提前准备演示数据和稳定网络环境

### 9.4 项目范围失控

风险说明：

- 功能越做越多，主链路迟迟不能完成

应对方案：

- 严格先做主链路
- 非核心功能放到第五阶段以后
- 没有完成主链路前，不开始额外优化项
