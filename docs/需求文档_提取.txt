企业知识库智能助手》需求文档 V1
1. 项目概述
项目名称：企业知识库智能助手
项目定位：面向企业内部场景的知识库问答与管理系统。系统支持企业文档上传、解析、切片、向量化、知识库管理与智能问答，帮助员工快速获取制度、流程、规范等信息，并提供引用溯源与效果反馈能力。
项目目标：
搭建一个可落地的企业内部 AI 知识问答平台
实现从文档入库到问答返回的完整 RAG 链路
保证问答结果可追溯、可管理、可优化
支持基础权限隔离，避免非授权用户访问知识内容
通过 Docker Compose 提升项目部署与演示效率
2. 建设范围
V1 范围：
用户登录与角色管理
知识库管理
知识库成员管理
文档上传与管理
文档解析、切片、向量化
基于知识库的智能问答
引用溯源
会话历史
答案反馈
失败问题记录
基础后台统计
Docker Compose 部署
V2 预留：
热门问题统计大盘
重排序 rerank
混合检索
异步任务队列
更细粒度权限控制
3. 用户角色
管理员：
管理用户
创建和管理知识库
管理知识库成员
上传和删除文档
查看问答记录、反馈记录、失败问题
查看系统统计信息
普通用户：
登录系统
查看自己被授权的知识库
在已授权知识库内发起问答
查看自己的会话历史
对答案进行反馈
4. 业务规则
系统仅允许基于已授权知识库进行问答
普通用户不能访问未加入的知识库
文档上传后必须完成解析、切片、向量化，才能参与检索
问答结果必须附带引用来源
当未检索到有效内容时，系统应明确提示“未检索到相关知识内容”
反馈和失败问题记录要与具体问答关联，便于后续优化
5. 功能需求
5.1 用户登录
功能说明：用户通过账号密码登录系统，获取访问权限。
详细需求：
提供登录页，输入用户名和密码
登录成功后返回用户基本信息、角色信息、Token
登录失败时提示用户名或密码错误
支持退出登录，退出后清除本地登录态
未登录用户不可访问业务页面
Token 过期后需重新登录
输入：
用户名
密码
输出：
登录结果
用户信息
角色信息
访问 Token
5.2 用户与角色管理
功能说明：管理员维护系统用户，并区分管理员与普通用户角色。
详细需求：
管理员可新增用户
管理员可编辑用户名、状态、角色
管理员可重置用户密码
管理员可禁用用户
普通用户不可访问用户管理页面
系统角色至少包含 管理员、普通用户
关键规则：
用户名唯一
禁用用户不可登录
普通用户不能提升自己权限
5.3 知识库管理
功能说明：管理员创建和维护知识库，用于承载某一类企业文档。
详细需求：
管理员可创建知识库
创建时需填写知识库名称、描述
管理员可查看知识库列表
管理员可编辑知识库名称和描述
管理员可删除空知识库或手动确认删除已有文档的知识库
普通用户只能看到自己被授权加入的知识库
知识库详情页需展示文档数量、成员数量、创建时间
关键规则：
知识库名称在系统内建议唯一
删除知识库时应同步删除关联文档元数据和向量数据
普通用户无权创建或删除知识库
5.4 知识库成员管理
功能说明：管理员为知识库分配可访问成员，实现指定成员可见。
详细需求：
管理员可向知识库添加成员
管理员可移除成员
管理员可查看当前成员列表
普通用户仅能访问自己加入的知识库
问答接口必须校验当前用户是否属于该知识库成员
关键规则：
未加入成员列表的用户不能查看知识库详情
未加入成员列表的用户不能发起该知识库问答
同一用户不能重复加入同一知识库
5.5 文档上传与管理
功能说明：管理员向知识库上传业务文档，作为知识内容来源。
支持格式：
PDF
DOCX
Markdown
TXT
详细需求：
管理员在知识库下上传文档
上传后记录文件名、类型、大小、上传人、上传时间
文档列表页展示处理状态，包括待处理、处理中、成功、失败
支持删除文档
支持查看文档解析结果概要，如 chunk 数量、页数、处理耗时
同名文件允许上传，但需通过文档 ID 区分
关键规则：
文档必须归属某个知识库
删除文档时需同步删除其切片记录和向量记录
解析失败的文档不能进入问答检索
5.6 文档解析
功能说明：系统将原始文档解析为可处理的纯文本内容。
详细需求：
PDF 需提取正文文本和页码信息
DOCX 需提取段落文本
Markdown 需保留标题层级信息
TXT 按自然段读取
解析后保存原始文本结果或结构化片段信息
对无法解析的文档记录失败原因
关键规则：
解析结果应尽量保留章节结构
解析异常时更新状态为失败并记录错误日志
页码信息优先保留，用于后续引用溯源
5.7 文本切片
功能说明：将解析后的长文本拆分为适合向量化和检索的知识片段。
详细需求：
按标题、自然段进行优先切分
每个 chunk 控制在约 300-500 中文字
相邻 chunk 保留 50-100 字重叠内容
为每个 chunk 生成序号、来源文档、页码、字符数等元数据
切片结果写入数据库保存
关键规则：
切片不能随意打断表格或列表结构
切片过短时可合并相邻段落
切片必须保留与原文档的映射关系
5.8 向量化与入库
功能说明：系统对每个文本切片生成向量，并写入向量数据库。
详细需求：
使用 embedding 模型对 chunk 文本生成向量
向量写入 Qdrant
同时保存向量关联的 payload 信息
payload 至少包含知识库 ID、文档 ID、chunk ID、页码、文件名
入库成功后更新文档状态为可用
关键规则：
向量库仅存储系统上传文档产生的数据
向量入库失败时应保留失败原因
向量与数据库中的 chunk 记录必须一一对应
5.9 智能问答
功能说明：用户在指定知识库内输入问题，系统基于 RAG 返回答案。
详细需求：
用户选择一个已授权知识库后发起提问
系统将问题转为检索向量
按知识库范围过滤召回相关 chunk
取 TopK 结果拼接上下文
调用大模型生成答案
返回答案内容、引用来源、回答时间
支持多轮会话，同一会话下连续提问
关键规则：
问答必须限制在当前知识库范围内
模型回答应优先依据检索到的上下文
若检索结果不足，应明确提示未找到可靠依据
问答过程需记录问题、答案、模型名、时间
5.10 引用溯源
功能说明：为每次回答提供依据来源，提升可解释性。
详细需求：
返回引用文档名
返回引用页码或 chunk 序号
返回引用片段摘要
前端支持点击查看引用信息
一次回答可附带多个引用来源
关键规则：
无引用的答案不应标记为高可信
引用必须与实际召回 chunk 对应
若文档无页码，则至少返回文件名和片段序号
5.11 会话历史
功能说明：记录用户历史问答，便于追溯和继续对话。
详细需求：
系统自动创建会话
支持查看会话列表
支持查看单个会话下的全部消息
支持删除个人会话
会话标题可由首个问题自动生成
关键规则：
普通用户仅能查看自己的会话
管理员如需查看全局会话，应作为后台统计能力单独开放
删除会话只删除会话展示数据，不影响知识库内容
5.12 答案反馈
功能说明：用户对问答结果进行评价，用于后续优化。
详细需求：
用户可对答案进行点赞或点踩
点踩时可填写原因，如“答非所问”“引用不准确”“未解决问题”
系统保存反馈时间、反馈人、关联消息
管理员可在后台查看反馈记录
关键规则：
同一用户对同一条回答只允许保留一条当前反馈
反馈数据用于后续调优，不直接修改原答案
5.13 失败问题记录
功能说明：记录未命中或效果较差的问题，形成优化闭环。
详细需求：
当系统未检索到相关内容时自动记录失败问题
当用户对答案点踩时可同步记入失败问题池
记录问题内容、所属知识库、失败类型、时间
管理员可查看失败问题列表
失败类型至少包括“无检索结果”“回答质量差”“引用不足”
关键规则：
失败问题记录需关联具体知识库
后续可用于补充文档、优化切片和提示词
5.14 后台统计
功能说明：管理员查看系统整体使用情况。
V1 统计项：
知识库数量
文档数量
已处理文档数量
问答总次数
反馈总数
失败问题总数
关键规则：
V1 以基础统计为主
热门问题排行放入 V2
6. 非功能需求
易用性：页面清晰，管理操作路径简单
性能：单次普通问答响应时间尽量控制在可接受范围内
可维护性：业务服务与 AI 服务职责分离
可扩展性：支持后续增加更多模型、更多文档格式、更多检索策略
安全性：接口需做登录校验和权限校验
可部署性：支持 Docker Compose 一键启动
